Security
Authentication & security model
JWT stateless auth, Argon2 password hashing, XSS prevention, and a dual guest/registered user model.
🔐
Argon2 Password Hashing
Winner of the 2015 Password Hashing Competition. Memory-hard, GPU-resistant. Automatically upgrades old hashes on verify via passlib's deprecated="auto" flag.
CryptContext(schemes=["argon2"]) # → $argon2id$v=19$m=65536... # memory-hard: 64 MB RAM per hash
🎫
JWT — HS256, 24h expiry
Tokens carry the user UUID as sub. The get_optional_user dependency decodes on every request; any failure (expired, tampered) silently returns None — guest mode, not a 401.
{ "sub": "3fa8...uuid", "iat": 1704067200, "exp": 1704153600 } # → signed with JWT_SECRET_KEY env var
👤
Optional Auth Design
Every POST endpoint accepts either a Bearer token or no token. Guest posts store a free-text author_name with author_id = NULL. The wall is always open — auth is never a barrier.
if user: author_id = user.id author_name = user.username else: author_id = None # guest author_name = payload.author_name
🧹
XSS Prevention
All user-generated content is passed through escapeHtml() before any DOM insertion. No raw innerHTML from API strings anywhere in the frontend code.
// <script>alert(1)</script> as author_name // becomes in DOM: // &lt;script&gt;alert(1)&lt;/script&gt; // → inert text, never executed
🗑️
Cascade Delete Strategy
When a post is deleted, all its comments cascade-delete (ON DELETE CASCADE). When a user is deleted, their posts and comments remain but author_id is set to NULL — content is preserved, identity is removed.
Post.author_id FK → users.id ON DELETE SET NULL ← user gone, post stays Comment.post_id FK → posts.id ON DELETE CASCADE ← post gone, comments go

Data access
Cursor-based pagination
Keyset pagination — no OFFSET, stable results, O(log n) instead of O(n) on deep pages.
Why not OFFSET?
OFFSET 1000
PostgreSQL must scan and discard 1000 rows. Gets slower as users page deeper. Unstable if new posts arrive.
Cursor pagination
Jumps directly to the indexed position. O(log n). Stable even when new posts are added mid-scroll.
How the cursor works
The cursor is a base64url-encoded JSON blob containing the last item's created_at and id:
// cursor encodes: {"created_at":"2026-01-15T10:30Z", "id":"abc-123-..."} // SQL WHERE clause: WHERE (created_at < '2026-01-15T10:30Z') OR (created_at = '2026-01-15T10:30Z' AND id < 'abc-123-...') ORDER BY created_at DESC, id DESC
